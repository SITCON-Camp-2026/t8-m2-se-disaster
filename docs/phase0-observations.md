# Phase 0 Observations

## v1 優先服務的使用者

- [ ] 回報者
- [x] 資訊整理者
- [ ] 行動者

原因：

目前的工作台最適合協助資訊整理者看懂原始資訊、候選分類、待確認處與不可派工原因。回報者需要的是更簡單的輸入流程；行動者需要的是已確認任務與安全指示，這些都超出 Phase 0 的安全範圍。

## 這個使用者最想完成什麼

資訊整理者最想快速把混亂原始資訊分成幾類：可能是求助、地點狀態、支援或公告；同時留下哪些地方不能判斷，讓下一位協作者知道要先查什麼。

## 這個使用者最怕什麼錯誤

最怕把未確認資訊看成已確認事實，或把看起來像任務的文字直接派給行動者。像「有人說」「截圖」「疑似」「不知道下午還有沒有」這類線索，都可能讓整理結果看起來比原文更可靠。

## 我今天暫時不處理的需求

- 不做回報者輸入表單。
- 不做行動者任務清單或派工流程。
- 不做地圖、路線、真實地址或外部資料查詢。
- 不把候選分類轉成正式資料模型。

## 我不採用的 AI 建議

AI 建議：

把所有事件直接整理成任務、地點狀態或公告，並讓畫面看起來像已完成分類。

不採用原因：

原始資訊多數仍是 `needs_review` 或 `unverified`，AI 的候選分類只能協助閱讀，不能取代人工確認。若直接做成正式任務，會讓使用者誤以為資料已經可信。

## 必須保留人工確認的判斷

- 來源是否可靠，是否只是轉述或留言。
- 時間是否仍有效，資料更新時間不等於事件發生時間。
- 地點是否足夠清楚，不能把模糊描述補成地址。
- 當事人是否同意公開資訊或接受協助。
- 是否真的可以派人，或只是需要先查證。

## 今天明確不做什麼

- 不新增後端、資料庫、外部 API 或 runtime LLM。
- 不查真實地圖、新聞、政府資料或社群來源。
- 不新增真實個資、電話、地址或災情資料。
- 不把 `needs_review` / `unverified` 顯示成已確認。
- 不讓 AI 自動做救災判斷。

## 我仍不確定的地方

- 候選分類的類型是否足夠資訊整理者使用。
- 哪些欄位應該成為正式資料格式，哪些只適合留在備註。
- 什麼情況下可以從候選分類進到下一步人工確認。
- 「不可派工原因」是否需要更細的標籤，方便之後交接。

## 我們做出了什麼？

- 一個可從首頁進入的 Phase 0 整理工作台。
- 每筆原始資訊會顯示來源、查核狀態、候選分類、整理警示與不可派工原因。
- 分類表的「查看」可以切換事件並回到上方詳細卡。

## 哪些資料讓 agent 很容易整理？

- 有明確物資、地點或公告線索的資料，例如活動中心、雨鞋、公告、道路封閉等。
- 但這些仍只能當候選分類，不能直接變成已確認事實。

## 哪些資料讓 agent 亂補或誤判？

- 地點模糊的資訊，像「老雜貨店後面」「老街口那邊」「附近」。
- 第三方轉述的資訊，像家屬來電、群組留言、截圖。
- 時間不清楚或可能過期的資訊，像昨天名單、下午是否還有物資。

## 哪些資訊不能直接變成任務？

- 沒有清楚地點、時間、當事人意願或現況確認的資訊。
- 原文說「不要再派人」但原因未明的資訊。
- 由第三方轉述、無法確認當事人目前狀況的資訊。

## 哪些資訊的操作者不是當事人？

- 社群貼文、群組留言、截圖、家屬來電、志工代轉述都可能不是當事人直接回報。
- 這類資料需要人工確認，不適合直接派工。

## 我們剛剛偷偷假設了什麼？

- 假設關鍵字可以幫忙分出候選類型。
- 假設資料更新時間可以輔助排序，但不能代表事件仍有效。
- 假設資訊整理者可以理解「候選結果」不是正式答案。

## 如果要讓這個工具變可靠，下一步需要什麼？

- 更清楚的資料格式。
- 人工確認角色與確認紀錄。
- 對「可派工」「不可派工」「需要補問」的狀態定義。

## 我們需要更清楚資料格式的地方

- 事件時間、資料更新時間、來源類型、查核狀態要分開。
- 地點線索要能標示「模糊」「不可公開」「需要補問」。
- 需求、資源、公告、支援者不能混在同一個欄位裡。

## 我們需要人工確認的地方

- 是否真的有需求。
- 是否仍有效。
- 是否能公開地點或當事人資訊。
- 是否可以派人，或只能先補問。

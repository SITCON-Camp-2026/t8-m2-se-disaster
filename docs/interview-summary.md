# Main Agent 訪談彙整

> 這份文件由 Codex 依三個 persona 訪談紀錄產生草稿。請學員自行刪除、修改或補充，並保留人類確認痕跡。

## 三個 persona 的共同需求

- 三個角色都需要清楚知道資訊是否已確認。這是使用者明確說的。
- 三個角色都在意時間、地點、來源是否完整。這是使用者明確說的。
  

## 三個 persona 之間的衝突需求

- 回報者想快速送出不完整資訊；資訊整理者需要足夠欄位判斷能不能相信。
- 行動者想直接知道能不能出發；資訊整理者只能提供候選分類與人工確認狀態。
- 回報者希望系統不要否定他的回報；行動者需要很強的警示，避免把不完整回報當成任務。

## Phase 0 原型中被推翻的假設

- 原本假設：
  - 「候選分類」加上「不可派工」就足夠避免誤解。
- 被什麼回饋推翻：
  - 資訊整理者和行動者都提到「任務候選」但又「不可派工」需要多看才懂。
  - 行動者明確問：哪個標示代表「我現在絕對不能出發」？

## 最容易造成誤導的風險

- 「高：仍需確認」可能讓人以為資料比較可靠。
- 「任務候選」可能讓行動者誤以為這已經是任務。
- 回報者的轉述可能被畫面整理得太像已確認資訊。
- 過期、衝突或缺地點的資訊，如果警示不夠明顯，仍可能讓人白跑或做錯事。

## 可以考慮的設計方向

> 這裡只是可考慮方向，不是標準答案。

- 把「候選分類」和「可行動狀態」分成更清楚的兩個欄位。
- 將「不可派工」改成更明確的警示，例如「目前不可出發」或「需人工確認」。
- 對轉述、截圖、社群貼文加上更明顯的「不是親眼確認」提示。
- 降低「高 / 中 / 低」這種容易被誤解成可信度的語言。

## 需要人類決策的取捨

- 取捨 1：
  - 選項 A：v1 繼續優先服務資訊整理者，強化人工確認與交接。
  - 選項 B：v1 轉向行動者，提供更直接的出發與不可出發提示。
  - 我目前傾向：選項 A。
  - 理由：目前資料仍是 Phase 0 原始資訊，多數是 `needs_review` 或 `unverified`，不適合讓行動者直接使用。

- 取捨 2：
  - 選項 A：讓回報者可以送出不完整資訊。
  - 選項 B：要求回報者補齊欄位才可送出。
  - 我目前傾向：暫不處理回報者輸入。
  - 理由：Release 01 是需求分析，且目前 prototype 的核心是整理既有原始資訊，不是建立回報流程。

## 不應該讓 AI 自動決定的事情

- 資訊是否已確認。
- 是否可以派工或出發。
- 模糊地點的真實地址。
- 當事人是否同意公開資訊。
- 轉述內容是否等同現場確認。

## 我採用的 AI 整理

- 採用三個 persona 對「候選結果容易被誤認成任務」的共同提醒。
- 採用「v1 優先服務資訊整理者」的方向，但保留人類確認痕跡。
- 採用「高 / 中 / 低可能造成誤解」作為需要檢查的設計風險。

## 我不採用的 AI 建議

- 建議：
  - 直接把回饋整理成完整功能清單。
  - 不採用原因：Release 01 的目標是需求分析與取捨，不是直接進入產品設計或實作。
- 建議：
  - 讓行動者也能用這個畫面判斷下一步。
  - 不採用原因：目前資料未確認，讓行動者使用會放大誤判風險。

## 我自己補充的判斷

- v1 可以繼續服務資訊整理者，但要避免使用會被行動者誤讀的語言。
- 「候選分類」應該更像整理標籤，不應該看起來像任務狀態。
- 人工確認不只是 UI 標示，還需要留下「誰確認、確認了什麼、仍缺什麼」的紀錄格式。

## 我仍不確定的地方

- 「任務候選」是否應該改名，避免行動者誤解。
- 資訊整理者需要多細的不可派工原因，才足夠交接。
- 是否應該在 v1 隱藏行動者可能誤讀的欄位。

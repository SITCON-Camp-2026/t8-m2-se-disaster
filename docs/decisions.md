# 需求取捨決策

> 這份文件由 Codex 產生草稿，最後必須由學員刪除、修改或補充。請不要讓 AI 替你做最後決策。

## 我決定 v1 優先服務的使用者

- [ ] 回報者
- [x] 資訊整理者
- [ ] 行動者

原因：

目前 Phase 0 prototype 已經聚焦在整理原始資訊、標示候選分類與列出待確認處。因此 v1 先服務資三個 persona 的回饋都指出：未確認資訊很容易被誤解成可行動任務。訊整理者，比直接服務行動者安全，也比新增回報流程更符合現有原型。

## 這個使用者最想完成什麼

資訊整理者最想把混亂的原始資訊整理到下一位協作者能理解：看出可能類型、來源、查核狀態、缺漏脈絡，以及為什麼現在不可派工。

## 這個使用者最怕什麼錯誤

- 把模糊資訊直接變成可以行動的任務。
- 分不清楚哪些是原文，哪些是 AI 推測。
- 候選分類被行動者誤解成已確認任務。

## 我今天暫時不處理的需求

- 不做回報者輸入表單。
- 不做行動者任務清單。
- 不做派工流程。
- 不做個人救援次數積分排行榜。
- 不做地圖、路線或真實地址查詢。
- 不新增後端、資料庫或外部 API。

## 我不採用的 AI 建議

- AI 建議：
  - 讓行動者也可以直接用分類結果判斷是否出發。
  - 不採用原因：目前資料仍是 Phase 0 原始資訊，多數需要人工確認，不能讓行動者直接使用。

- AI 建議：
  - 把 persona 回饋整理成完整功能清單。
  - 不採用原因：Release 01 是需求分析，不是實作階段；現在應該先保留取捨，不急著做功能。

## 新增需求評估：個人檔案與積分排行榜

- 原始想法：
  - 設立個人檔案資料。
  - 加入積分排行榜。
  - 依每個人的救援次數多寡給予積分。
- 我目前採用的部分：
  - 可以保留「匿名協作者代號」與「角色」，用來記錄誰做了整理、確認或修正。
  - 可以在操作紀錄中看見每位協作者做過哪些判斷。
- 我目前不採用的部分：
  - 不建立真實個人檔案。
  - 不用救援次數計分。
  - 不做救援者排行榜。
- 人類判斷理由：
  - 目前 v1 優先服務資訊整理者，不是行動者或派工管理者。
  - 救援次數需要真實任務完成紀錄與安全確認，不能從 Phase 0 原始資訊推算。
  - 排行榜可能讓使用者為了分數把未確認資訊當成任務，會放大誤判風險。

## 必須保留人工確認的判斷

- 資訊是否已確認。
- 轉述者是否等於當事人或現場目擊者。
- 時間是否仍有效。
- 地點是否足夠清楚，且是否可以公開。
- 是否真的可以派工或出發。
- 協作者紀錄是否可以公開，以及是否會被誤解成救援績效。

## 今天明確不做什麼

- 不修改 `src/`。
- 不新增真實資料、真實地圖、真實地址、電話或個資。
- 不查外部社群、地圖、政府資料或新聞。
- 不把 `needs_review` / `unverified` 顯示成已確認。

## 我仍不確定的地方

- 「任務候選」是否需要改名，避免被行動者誤解。
- 「高 / 中 / 低」是否應該移除，避免看起來像可信度。
- v1 是否需要明確定義「整理完成，可以交給人類確認」的條件。
- 不可派工原因要細到什麼程度，才足夠下一位協作者接手。
- 如果未來要顯示貢獻統計，哪些統計不會鼓勵不安全行動。

## 人類確認痕跡

- 我採用的 AI 建議：
  - v1 先服務資訊整理者。
  - 保留候選分類，但要更清楚標示不能派工。
- 我不採用的 AI 建議：
  - 不讓行動者直接依照目前畫面出發。
  - 不把訪談結果立刻轉成功能清單。
- 我自己補充的判斷：
  - v1 應該先降低資訊整理者誤判風險，再考慮服務回報者或行動者。
  - 若未來要服務行動者，必須先有明確的人工確認狀態與安全邊界。
